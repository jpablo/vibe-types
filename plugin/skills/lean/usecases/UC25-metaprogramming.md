# Metaprogramming and Syntax Extension

## The constraint

Extend the language safely at compile time. New syntax, custom tactics, and code generation are type-checked after expansion — no ill-typed code can be generated by macros or elaborators.

## Feature toolkit

- [→ T17-macros-metaprogramming](../catalog/T17-macros-metaprogramming.md) — `syntax`, `macro_rules`, and `elab` for compile-time metaprogramming.
- [→ T39-notation-attributes](../catalog/T39-notation-attributes.md) — `notation`, attributes, and compiler options for lightweight extensions.

## Patterns

### Pattern A — Custom notation

```lean
notation:65 lhs " ⊕ " rhs => HAdd.hAdd lhs rhs  -- custom operator

infixr:70 " ∘ " => Function.comp  -- function composition operator

#eval (· + 1) ∘ (· * 2) <| 3  -- 7
```

### Pattern B — Syntax macro for DSL

```lean
-- Define a small assertion DSL
syntax "check_that " term " is " term : term

macro_rules
  | `(check_that $actual is $expected) =>
    `(if $actual == $expected then IO.println "PASS" else IO.println "FAIL")

def main : IO Unit := do
  check_that (2 + 2) is 4     -- PASS
  check_that (2 + 2) is 5     -- FAIL
```

### Pattern C — Custom tactic

```lean
-- A tactic that applies `constructor` then tries `simp` on all goals
syntax "auto_construct" : tactic

macro_rules
  | `(tactic| auto_construct) =>
    `(tactic| constructor <;> simp)

example : 1 + 1 = 2 ∧ 2 + 2 = 4 := by auto_construct
```

### Pattern D — Elaboration-time code generation

```lean
-- Generate a ToString instance automatically
-- (This is what `deriving ToString` does internally)
elab "derive_toString " ident:ident : command => do
  -- Access the environment, inspect the type, generate the instance
  -- All generated code is type-checked by the kernel
  sorry -- simplified
```

## Tradeoffs

| Pattern | Strength | Weakness |
|---------|----------|----------|
| Notation | Simple; good for math-like syntax | Limited to expression transformations |
| Macros | Hygienic; purely syntactic | No access to type information |
| Custom tactics | Powerful proof automation | Requires understanding the tactic framework |
| Elaborators | Full power; access to types and env | Complex API; steep learning curve |

## When to use which feature

- **Mathematical notation** (operators, special symbols) → `notation` or `infixl`/`infixr`.
- **Syntactic sugar** (DSLs, assertion helpers) → `macro_rules`.
- **Proof automation** (domain-specific tactics) → custom tactics via `elab`.
- **Code generation** (deriving instances, boilerplate) → elaboration with `Lean.Elab` API.
